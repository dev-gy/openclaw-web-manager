### CRIT-001: 활동 로그 피드(F-006)의 데이터 소스 아키텍처가 미정의

- 심각도: 높음
- 대상: F-006 활동 로그 피드 -- "로그 소스" 항목
- 지적: 스펙은 "useAgentActivity의 WebSocket onMessage에서 session.started / session.updated / session.ended 이벤트를 캡처"하라고 명시하지만, 현재 `useAgentActivity` 훅의 `onMessage` 콜백은 agents 상태 배열을 갱신하는 역할만 수행하며, 개별 이벤트를 외부로 노출하는 인터페이스가 없다. 훅의 반환 타입에는 `agents`, `loading`, `error`, `wsConnected`, `refresh`, `totalSessions`, `activeSessions`만 존재한다. 로그 피드 컴포넌트가 WebSocket 이벤트를 수신하려면 (1) `useAgentActivity` 훅의 반환값에 이벤트 히스토리 배열을 추가하거나, (2) 별도의 콜백/이벤트 emitter를 노출하거나, (3) 페이지 컴포넌트에서 별도 WebSocket 연결을 열어야 한다. 세 가지 중 어떤 방법을 사용할지 스펙에 명시되어 있지 않아 개발자가 해석의 여지 없이 구현할 수 없다.
- [근거: Read `hooks/useAgentActivity.ts` -- 반환 객체(206-214행)에 이벤트 로그 관련 필드 없음. `onMessage` 콜백(128-148행)은 `setAgents`만 호출하며 이벤트 히스토리를 저장하지 않음]
- 제안: `useAgentActivity`의 반환 객체에 `activityLogs: ActivityLogEntry[]`를 추가하고, `onMessage` 콜백 내에서 이벤트 발생 시 로그 배열에 push하는 방식을 스펙에 명시할 것. 또는 `onActivityEvent?: (event: ActivityLogEntry) => void` 콜백 옵션을 훅에 추가하는 방식도 가능. 어느 쪽이든 인터페이스 변경 범위를 명확히 정의해야 한다.

### CRIT-002: SVG + absolute positioned HTML 하이브리드 레이아웃의 좌표 계산 방식 미정의

- 심각도: 높음
- 대상: F-001 싱글 플로어 오피스 컨테이너 + F-003 에이전트 캐릭터 공간 배치
- 지적: 스펙은 "배경은 SVG, 에이전트/가구는 absolute positioned HTML"이라고 명시하면서, 바닥선은 "컨테이너 하단 20% 위치"라고 정의한다. 그런데 에이전트 캐릭터를 `position: absolute`로 바닥선 위에 배치하려면, SVG 좌표계와 HTML absolute 좌표계를 동기화해야 한다. 이에 대한 구체적 방법이 없다. 컨테이너 높이가 420-520px 범위에서 가변적이라면 바닥선의 실제 픽셀 y좌표도 가변적이고, SVG viewBox 좌표와 HTML top/bottom 값 사이의 매핑 로직이 필요하다. 현재 코드베이스에는 SVG-HTML 하이브리드 좌표 동기화를 수행하는 유틸리티나 패턴이 전혀 없다.
- [근거: Grep `position: absolute` in `pages/monitor/` -- 결과 0건. 기존 페이지에 SVG+absolute HTML 하이브리드 레이아웃 사례 없음. Read `pages/monitor/agents/+Page.tsx` -- 현재 레이아웃은 순수 Tailwind grid/flex 기반]
- 제안: 좌표 계산 전략을 명확히 지정할 것. 예를 들어 "(1) SVG는 순수 배경 장식(바닥선, 벽면)만 담당하고 viewBox를 컨테이너 크기에 맞춤, (2) 바닥선 y좌표는 CSS `bottom: 20%`로 계산, (3) 에이전트는 `bottom: 20%` 기준으로 `bottom` 값을 계산"처럼, SVG 좌표와 HTML 좌표의 매핑 방식을 구체적으로 기술해야 한다. 또는 SVG를 버리고 전부 CSS로 구현하는 대안도 고려할 것.

### CRIT-006: 에이전트 상태 변경 시 존 이동 애니메이션 미정의

- 심각도: 중간
- 대상: F-003 에이전트 캐릭터 공간 배치 + B-6 유저 플로우
- 지적: 유저 플로우(B-6)는 "상태 변경 (WebSocket) -> 에이전트 존 이동"이라고 명시한다. 그런데 에이전트가 한 존에서 다른 존으로 이동할 때의 시각적 전환이 정의되지 않았다. 세 가지 가능성이 있다: (1) 즉시 텔레포트 (현재 위치에서 사라지고 새 위치에 나타남), (2) 슬라이딩 애니메이션으로 가로 이동, (3) 페이드아웃+페이드인. 어떤 방식인지 명시되지 않으면, 에이전트가 갑자기 사라졌다 나타나는 것이 의도인지 버그인지 구분이 어렵다. 특히 `position: absolute`로 좌표 기반 배치를 사용하므로 CSS transition만으로 자연스러운 이동이 가능한지도 검증이 필요하다.
- [근거: Read `01-spec.md` F-003 전체 -- "에이전트 위치 계산" 섹션에 초기 배치만 정의되어 있으며, 존 간 전환 애니메이션 언급 없음]
- 제안: 존 이동 시 시각 전환 방식을 명시할 것. 구현 난이도를 고려하면 "CSS transition (left, top 속성에 300ms ease-out)"을 적용하여 자연스러운 슬라이딩이 가장 단순한 선택이다. 또는 "존 이동은 즉시 반영 (애니메이션 없음)"도 유효한 선택이지만 명시적으로 기술해야 한다.

### CRIT-007: 사이드 패널(F-010) 열림 시 오피스 min-width 800px 제약과 충돌

- 심각도: 중간
- 대상: F-001 + F-010
- 지적: F-001은 오피스 컨테이너의 `min-width: 800px`을 정의하고, F-010은 사이드 패널이 열리면 오피스 가로 폭이 `calc(100% - 320px)`로 줄어든다고 명시한다. 사이드바(w-60 = 240px)가 존재하는 상태에서 1280px 화면이라면, 콘텐츠 영역은 약 1040px. 여기서 사이드 패널 320px를 빼면 오피스는 720px가 되어 `min-width: 800px` 미만이 된다. 이 경우 오피스가 800px를 유지하면서 가로 스크롤이 발생하는지, 아니면 min-width 제약을 무시하는지 불명확하다. 스펙은 "1280px 미만에서는 오버레이 모달로 표시"라고 하지만, 정확히 1280px일 때도 문제가 발생한다.
- [근거: Read `layouts/Sidebar.tsx` -- 사이드바 폭 `w-60` (240px). Read `01-spec.md` F-001 "min-width: 800px" + F-010 "calc(100% - 320px)" + "1280px 미만 화면에서는 사이드 패널 대신 오버레이 모달"]
- 제안: 사이드 패널 열림 시 오피스 `min-width`를 제거하거나 축소하는 규칙을 추가하거나, 사이드 패널의 오버레이 모달 전환 기준을 "1280px 미만"이 아니라 "오피스 가용 폭이 800px 미만이 될 때"로 변경할 것.

### CRIT-010: 가구 SVG 14종의 인라인 렌더링이 DOM 노드를 과도하게 증가시킬 수 있음

- 심각도: 낮음
- 대상: F-002 존별 가구 SVG 세트
- 지적: 가구 14종을 모두 인라인 SVG로 렌더링하면, 가구 1개당 평균 5-10개의 SVG 요소(rect, circle, path, line 등)가 필요하다고 가정할 때 70-140개의 SVG DOM 노드가 추가된다. 여기에 에이전트 아바타 SVG(1개당 약 15개 노드 -- AgentAvatar.tsx 확인), 말풍선, 바닥선/벽면까지 합치면 페이지 전체에 수백 개의 SVG DOM 노드가 생긴다. React 리렌더링 시 이 모든 노드를 diff해야 하므로 WebSocket 이벤트로 잦은 상태 갱신이 발생하면 성능 저하 가능성이 있다.
- [미확인: 실제 성능 영향은 브라우저와 에이전트 수에 따라 다르므로 프로파일링 없이 단정 불가. 다만 SVG 인라인 렌더링의 DOM 증가 문제는 일반적으로 알려진 이슈 -- LogRocket Blog "guide-svgs-react" 참고]
- 제안: (1) 가구 SVG는 상태 변경과 무관하므로 `React.memo`로 감싸서 불필요한 리렌더링을 방지하라는 성능 가이드를 스펙에 추가, (2) 또는 가구 SVG를 하나의 통합 SVG sprite로 그리는 대안을 언급할 것.

### CRIT-011: 에러 상태(B-8)에서 "재시도 버튼"의 동작이 F-007 헤더의 "새로고침 버튼"과 중복

- 심각도: 낮음
- 대상: B-8 상태 전이도 + F-007 오피스 헤더
- 지적: 에러 상태에서 "빨간 배너: 재시도 버튼"이 표시되고, 헤더에도 항상 "새로고침 버튼"이 존재한다. 두 버튼이 같은 `refresh`(=`fetchAgents`) 함수를 호출한다면 기능적으로 중복이다. 유저 관점에서 "재시도"와 "새로고침"의 차이가 불명확하다. 또한 에러 배너의 구체적 스타일(높이, 텍스트 크기, 배너 위치)이 Section A의 기능 정의에는 없고 B-8 상태 전이도에만 자연어로 기술되어 있어, 시각적 기대 상태(Section C)와 교차 검증하기 어렵다.
- [미확인: "재시도"와 "새로고침"이 의도적으로 구분된 것인지 확인 불가]
- 제안: 에러 배너의 재시도 버튼 동작을 명시하고, 헤더 새로고침과의 관계를 정의할 것. 예: "에러 배너의 재시도 = fetchAgents() + 에러 배너 닫기. 헤더 새로고침 = fetchAgents()만 호출"처럼 구분하거나, 에러 배너에 재시도 버튼 대신 "새로고침 버튼 사용 안내"를 표시하는 방안도 고려.
